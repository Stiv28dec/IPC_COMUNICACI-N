¿Qué es IPC?

La Comunicación Entre Procesos (Inter-Process Communication) es el conjunto de mecanismos que proporciona el sistema operativo para que procesos independientes puedan intercambiar datos y sincronizar sus acciones. Cada proceso tiene su propio espacio de memoria aislado, por lo que el kernel actúa como intermediario.

// Mecanismos IPC
Los 5 Métodos Principales

Cada mecanismo tiene características únicas de velocidad, complejidad y alcance. La elección depende del caso de uso específico.

// 01 · PIPES
Pipes
Anónimos
Los pipes son canales de comunicación unidireccionales creados por el kernel. Funcionan como un buffer circular FIFO: el proceso escritor envía datos al descriptor fd[1] y el proceso lector los consume desde fd[0].

Son ideales para procesos padre-hijo. El buffer del kernel tiene capacidad limitada (típicamente 64KB), y las operaciones se bloquean automáticamente si el buffer está lleno (escritura) o vacío (lectura).
// Crear y usar un pipe int fd[2]; pipe(fd); // crea el canal write(fd[1], "hola", 4); // escribe read(fd[0], buf, 4); // lee

// 02 · MESSAGE QUEUES
Colas de
Mensajes
Las colas de mensajes permiten enviar paquetes de datos con un tipo y prioridad asociados. El kernel mantiene la cola internamente, permitiendo comunicación asíncrona entre procesos sin necesidad de parentesco.

Los mensajes se reciben por prioridad descendente y luego por orden FIFO dentro de la misma prioridad, lo que los hace ideales para sistemas con tareas de distinta urgencia.
// POSIX Message Queue mqd_t mq = mq_open("/cola", O_RDWR); mq_send(mq, mensaje, 64, 5); // prio=5 mq_receive(mq, buf, 64, &prio);

// 03 · SHARED MEMORY
Memoria
Compartida
El mecanismo más rápido de IPC: múltiples procesos mapean la misma región de memoria física en sus espacios de direcciones virtuales. El acceso es directo, sin intervención del kernel en cada lectura/escritura.

⚠ Peligro: Sin sincronización, dos procesos que escriben simultáneamente causan una condición de carrera (race condition) corrompiendo los datos. La solución es usar semáforos.
// POSIX Shared Memory int fd = shm_open("/shm", O_CREAT|O_RDWR, 0644); ftruncate(fd, 4096); void* ptr = mmap(NULL, 4096, PROT_READ|PROT_WRITE, MAP_SHARED, fd, 0);

// 04 · SEMAPHORES
Semáforos
Los semáforos son variables enteras que controlan el acceso a recursos compartidos mediante dos operaciones atómicas: wait(S) decrementa el contador (bloquea si S=0) y signal(S) lo incrementa (desbloquea procesos en espera).

El semáforo binario (mutex) solo puede valer 0 o 1 y garantiza exclusión mutua. El semáforo contador gestiona N instancias de un recurso.
// Semáforo binario POSIX sem_t mutex; sem_init(&mutex, 0, 1); // S=1 sem_wait(&mutex); // wait(S): S-- /* sección crítica */ sem_post(&mutex); // signal(S): S++

// 05 · SOCKETS
Sockets
TCP / UNIX
Los sockets son el único mecanismo que permite IPC entre procesos en máquinas distintas a través de la red. Modelan una conexión bidireccional como un par de descriptores de archivo.

TCP (SOCK_STREAM) garantiza entrega ordenada y confiable. UDP (SOCK_DGRAM) sacrifica confiabilidad por velocidad. UNIX sockets operan localmente a través de un archivo del sistema de archivos.
// Servidor TCP básico int s = socket(AF_INET, SOCK_STREAM, 0); bind(s, &addr, sizeof(addr)); listen(s, 10); int c = accept(s, &cli, &len); recv(c, buf, 1024, 0);

Comparación de Mecanismos

Guía rápida para elegir el mecanismo IPC correcto según las necesidades del sistema.

Mecanismo Velocidad Dirección Persistencia Red Sincronización Uso Típico
Pipe MEDIA Unidireccional No (proceso vive) No Automática (bloqueo) shell pipes, padre-hijo
Cola de Mensajes MEDIA Bidireccional Sí (en kernel) No Prioridades tareas con prioridad
Memoria Compartida ALTA ⚡ Bidireccional Sí (shm_unlink) No Manual (semáforos) bases de datos, buffers
Semáforos ALTA N/A (sync) Sí (sem_unlink) No Nativa mutex, exclusión mutua
Sockets MEDIA Bidireccional Sí (red) SÍ 🌐 Protocolo TCP/UDP microservicios, APIs
⚡
Máximo Rendimiento
Usa Memoria Compartida con semáforos para la comunicación más veloz entre procesos en la misma máquina. Ideal para buffers de alta frecuencia.
shm + sem
⚖
Balance Simplicidad
Usa Colas de Mensajes cuando necesitas comunicación asíncrona con prioridades y sin la complejidad de sincronización manual.
mq_send / mq_recv
🌐
Comunicación en Red
Usa Sockets TCP/UDP cuando los procesos están en distintas máquinas o necesitas un protocolo estándar interoperable.
socket + connect

// Ciclo Completo
Cómo Funciona un
Pipe en el Kernel

Seguimiento paso a paso de cómo el kernel gestiona la comunicación a través de un pipe entre un proceso padre y su hijo.

01
pipe(fd)
El kernel crea dos descriptores de archivo: fd[0] (lectura) y fd[1] (escritura) apuntando al mismo buffer circular.
02
fork()
El proceso padre se bifurca. El hijo hereda los file descriptors abiertos, incluyendo los dos extremos del pipe.
03
write(fd[1])
El escritor llama a write(). El kernel copia los datos al buffer del pipe. Si está lleno, el proceso se bloquea.
04
read(fd[0])
El lector llama a read(). El kernel extrae datos del buffer FIFO. Si está vacío y hay escritores, bloquea y espera.
05
close()
Al cerrar todos los extremos de escritura, el lector recibe EOF en el siguiente read(). El pipe es destruido.
